## 2026-07-16 - Case-insensitive Substring Matching vs Regex in Hot Paths
**Learning:** Utilizing regular expression matches like `plan =~ ~r/enterprise|team/i` in hot paths such as upstream candidate selection and model routing introduces non-trivial CPU overhead from regular expression evaluation. Since these plan identifiers are simple alphanumeric strings, they can be matched significantly faster using case-insensitive substring checks such as `String.contains?(String.downcase(plan), "...")`. Furthermore, converting small lists to `MapSet` for membership checks inside loops is slower than basic linear list lookups.
**Action:** Avoid inline regex matching for basic substring or exact checks in performance-critical code. Prefer native string/binary functions and native collection operators for small collections (like list membership `in`) instead of constructing intermediate data structures like `MapSet`.
